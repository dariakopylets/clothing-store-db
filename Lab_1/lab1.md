# 1. Короткий виклад вимог
## Мета системи: Створення бази даних для зберігання та обробки інформації про клієнтів та замовлень в онлайн-магазині одягу.
Для цієї лабораторної роботи, ми обрали розробку бази даних для зберігання та обробки інформації про клієнтів та замовлень в онлайн-магазині одягу
Зацікавлена сторона: [Савченко] – представник ігрової платформи
Аналітик бази даних: [Копилець] – збір та документування вимог
## Потреби зацікавлених сторін:
-	Клієнт: Можливість реєстрації, перегляду каталогу товарів, формування кошика та оформлення замовлень.
-	Адміністрація магазину: Облік товарів, контроль замовлень, відстеження статусів оплати та управління брендами та категоріями.
## Дані для зберігання:
-	Особисті дані клієнтів (контакти, логін/пароль).
-	Каталог товарів з характеристиками (назва, ціна, модель).
-	Перелік брендів та категорій.
-	Історія замовлень (дати, статуси, загальна сума).
-	Деталі замовлень (які саме товари, якого розміру та кольору були придбані).
-	Інформація про оплати.
## Основні бізнес-правила:
1.	Оформити замовлення може тільки зареєстрований клієнт.
2.	Одне замовлення може містити декілька товарів.
3.	Кожен товар обов'язково належить до одного бренду та однієї категорії.
4.	Оплата прив'язується до конкретного замовлення.

# 3. Список сутностей з атрибутами та пояснення зв'язків
### Client (Клієнт)
-	client_id (PK): Унікальний ідентифікатор клієнта в базі даних.
-	login: Унікальне ім'я (логін) для входу в особистий кабінет.
-	password: Пароль для захисту облікового запису.
-	name: Ім'я користувача.
-	surname: Прізвище користувача.
-	phone_number: Контактний номер телефону.
-	email: Електронна пошта для розсилок та чеків.
-	address: Фізична адреса для доставки товарів.
-	birthday: Дата народження клієнта.
### Products (Товари)
-	product_id (PK): Унікальний код товару в системі.
-	brand_id (FK): Посилання на ідентифікатор бренду, до якого належить товар.
-	category_id (FK): Посилання на ідентифікатор категорії товару.
-	name: Назва товару.
-	price: Поточна вартість товару.
-	model: Назва моделі.
### Category (Категорія)
-	category_id (PK): Унікальний ідентифікатор категорії.
-	name: Назва групи товарів (наприклад, "Футболки", "Взуття").
### Brand (Бренд)
-	brand_id (PK): Унікальний ідентифікатор бренду.
-	name: Назва компанії-виробника.
-	country: Країна реєстрації бренду.
### Order (Замовлення)
-	order_id (PK): Унікальний номер замовлення.
-	client_id (FK): Посилання на клієнта, який оформив це замовлення.
-	date: Дата та час створення замовлення.
-	status: Поточний стан виконання (наприклад, "Обробляється", "Виконано").
-	price_at_the_moment: Фіксована ціна на момент оформлення замовлення.
### Cart (Вміст замовлення)
-	cart_id (PK): Унікальний ідентифікатор конкретного запису (позиції) в замовленні.
-	order_id (FK): Посилання на замовлення, до якого належить ця позиція.
-	product_id (FK): Посилання на товар, який було додано.
-	quantity: Кількість одиниць обраного товару.
-	price: Ціна товару.
-	color: Обраний колір товару.
-	size: Обраний розмір товару.
### Payment (Оплата)
-	payment_id (PK): Унікальний код платіжної транзакції.
-	order_id (FK): Посилання на замовлення, яке оплачується.
-	type: Метод оплати (наприклад, "Картка", "Google Pay").
-	status: Статус платежу (наприклад, "Успішно", "Очікується").
## Пояснення зв'язків:
-	client — order (1:N): Один клієнт може зробити багато замовлень, але кожне замовлення належить лише одному клієнту.
-	category — products (1:N): В одній категорії може бути багато товарів, але товар належить до однієї категорії.
-	brand — products (1:N): Один бренд випускає багато товарів, але конкретний товар має одного виробника.
-	order — cart (1:N): Одне замовлення складається з багатьох позицій (товарів у кошику), кожна позиція в таблиці cart посилається на одне замовлення.
-	products — cart (1:N): Один вид товару (наприклад, конкретна модель джинсів) може бути куплений у багатьох різних замовленнях (багато записів у cart), але запис у кошику посилається на конкретний товар.
-	order — payment (1:1): Одне замовлення оплачується лише один раз.

# 4. Припущення та обмеження
Під час розробки моделі були зроблені наступні припущення:
1.	Незмінність ціни в замовленні: Таблиця order містить власне поле price_at_the_moment. Це зроблено для того, щоб фіксувати ціну товару на момент покупки. Якщо ціна в таблиці products зміниться в майбутньому, це не повинно вплинути на історичні дані про вже оформлені замовлення.
2.	Обов'язкова реєстрація: Система не підтримує "гостьовий режим" оформлення замовлення; поле client_id в таблиці order є обов'язковим (NOT NULL).
3.	Валюта: Всі грошові поля (price) зберігаються в одній валюті системи за замовчуванням.
